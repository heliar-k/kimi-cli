"# 系统上下文概述\n\n## 1. 项目介绍\n\n### 1.1 项目名称与定位\n\n**kimi-cli** 是一个以本地开发工作流为中心的 AI 代理产品，定位为\"本机/局域网可部署的 AI Agent 工具链中枢\"。该系统将大语言模型（LLM）对话、工具调用、项目文件与会话历史有机整合，为开发者提供统一的 AI 辅助开发工作台。\n\n### 1.2 核心功能与价值\n\n#### 核心功能\n\n| 功能模块 | 描述 |\n|---------|------|\n| **CLI 交互** | 终端命令行界面，支持交互式配置向导，快速完成平台选择、鉴权与模型配置 |\n| **本地 Web 工作台** | React 构建的可视化工作台，提供会话管理、聊天界面、工具结果展示、Git 变更统计等 |\n| **会话管理** | 支持会话创建、持久化、回放、搜索、Fork（从指定轮次派生新会话） |\n| **工具执行体系** | 内置文件读写/搜索替换、Shell 命令执行、Web fetch/search、Todo 管理、多代理任务等工具 |\n| **多模型适配** | 支持 Kimi、OpenAI、Anthropic、Google GenAI 及 OpenAI 协议兼容端点 |\n| **MCP 扩展** | 支持 Model Context Protocol 服务器接入，扩展工具与资源访问能力 |\n| **SDK 支持** | 提供 kimi-sdk，便于二次开发与集成 |\n\n#### 业务价值\n\n- **提升研发效率**：将 AI 对话、代码阅读与工具执行整合到统一工作环境\n- **降低集成成本**：统一多家模型与工具链的接入方式，减少重复开发\n- **保障数据安全**：支持本地或局域网部署，敏感数据无需上传云端\n- **增强可控性**：工具调用审批机制，避免 AI 误操作\n\n### 1.3 技术特点概述\n\n```\n┌─────────────────────────────────────────────────────────────────┐\n│                      kimi-cli 技术架构特点                        │\n├─────────────────────────────────────────────────────────────────┤\n│  🏠 本地优先    │ 本机运行，数据不出域，支持局域网访问              │\n│  🔌 多模型适配  │ 统一抽象层支持 Kimi/OpenAI/Anthropic/Google 等   │\n│  📡 实时流式    │ WebSocket + JSON-RPC 事件流，支持历史回放         │\n│  🛠️ 工具扩展   │ 内置工具集 + MCP 协议扩展，支持审批机制           │\n│  📦 会话持久化  │ wire.jsonl 线性事件流，支持 Fork 与回溯           │\n│  🔐 安全控制    │ Token/Origin 校验、路径防护、敏感信息保护         │\n└─────────────────────────────────────────────────────────────────┘\n```\n\n---\n\n## 2. 目标用户\n\n### 2.1 用户角色定义\n\n#### 角色 A：个人开发者/工程师\n\n| 属性 | 描述 |\n|------|------|\n| 产品及其随附组件\"**。系统负责用户交互（CLI/Web）、会话与文件的安全管理、调用外部 LLM、编排工具与子进程，并向前端提供 API 与流式通道。\n\n**关键边界原则**：\n- ✅ 系统不托管模型，仅作为 LLM 服务的调用方\n- ✅ 系统不提供远程多租户 SaaS，仅支持本地/局域网部署\n- ✅ 系统不替代用户项目代码，仅在受控范围内读写\n\n### 3.2 1：本地优先架构\n\n**背景**：用户对数据隐私和离线能力有强烈需求\n\n**决策**：采用本地优先架构，所有核心功能在本机运行\n\n**影响**：\n- ✅ 数据不出域，满足安全合规要求\n- ✅ 支持离线配置和会话管理\n- ⚠️ 需要处理本地资源限制（存储、计算）\n- ⚠️ 多设备同步需要额外实现\n\n#### 决策 2：多模型统一抽象\n\n**背景**：不同 LLM 提供商 API 差异大，切换成本高\n\n**决策**：通过 kosong 层提供统一抽象，支持多模型适配\n\n**影响**：\n- ✅ 降低模型切换成本\n- ✅ 支持模型能力差异的优雅降级\n- ⚠️ 需要维护多模型协议适配\n- ⚠️ 部分模型特有功能可能无法完全利用\n\n#### 决策 3：事件流持久化\n\n**背景**：会话需要支持回放、搜索、Fork 等高级功能\n\n**决策**：采用 wire.jsonl 线性事件流持久化会话\n\n**影响**：\n- ✅ 支持精确的历史回放\n- ✅ 支持从任意轮次 Fork 新会话\n- ⚠️ 需要管理事件流文件大小\n- ⚠️ 需要处理事件序列一致性\n\n#### 决策 4：WebSocket 实时通信\n\n**背景**：Web 前端需要实时展示 AI 响应和工具执行状态\n\n**决策**：采用 WebSocket 实现双向实时通信\n\n**影响**：\n- ✅ 低延迟的实时交互体验\n- ✅ 支持服务端主动推送\n- ⚠️ 需要处理连接管理和重连逻辑\n- ⚠️ 需要考虑局域网访问的安全控制\n\n---\n\n## 6. 技术架构概述\n\n### 6.1
